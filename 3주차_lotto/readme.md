이전 과제에서 하나의 메소드에서 하나의 기능을 하지 못한 부분을 완벽하게 구성하지 못했었습니다. 그 때문에 이번 과제는 미비한 부분을 확실하게 구상하고 이에 더해 클래스를 명확하게 나누라는 포인트를 잘 구상해보기 위해 노력했습니다. 기능목록 구상 후 바로 구현에 들어가지 않고, 이틀 동안 설계를 진행했습니다. 기능목록을 자세하게 작성하고 목록 하나당 메소드를 적어 나갔고 그 메서드가 들어갈 클래스를 적어 나갔습니다. 하나의 한 기능만 해야 했기에 세부적으로 목록을 추가해야만 했습니다. 더 이상 쪼갤 수 없을 때까지 기능목록을 세분화하고 이에 맞는 메소드들을 모두 명시한 후 의존관계에 대해 고민했습니다. 공부하던 중 MVC 패턴을 알게 되었고 이번 과제에 적용하고자 큰 틀을 MVC로 구상했습니다. 아무것도 모른 채로 설계하지 않고 큰 틀이 있다 보니 클래스 간의 의존관계를 좀 더 명확하게 구상할 수 있었습니다.
이전 과제를 진행하면서 생성자를 사용할 때마다 불필요한 인스턴스 변수들이 생겨난다는 점을 알게 되었습니다. 이 부분을 공부하다가 정적 팩토리 메소드를 알게 되었고 이번 과제에서 연습을 많이 할 수 있었습니다. 리팩토링하는 과정에서 결과적으로 정적 팩토리 메소드를 꼭 써야 하는 이유를 찾지 못해 연습해보는 정도에 그쳤지만, 메모리를 생각하는 데 많은 도움이 되었습니다.

이전 과제에서는 사용해보지 못한 stream과 람다를 활용해보며 학습할 기회가 되었습니다. 먼저 설계를 마치고 구현해보니 이전에 힘들어했던 메서드 내 15줄 지침을 위한 수정 소요가 없었습니다. 조금 아쉬운 부분은 설계하는 시간이 오래 걸려 구현 후 리팩토링 하는 시간이 적었다는 것입니다. 내일의 과제는 시간 배분을 잘해서 후회 없이 과제를 완성해보고 싶습니다. 정적 팩토링 메소드를 잘 활용하여 메모리도 고려할 줄 아는 코드를 구현해보고 싶습니다.
